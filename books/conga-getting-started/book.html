Integration users, client IDs, and secrets

A step-by-step guide to creating integration users
and making your first authenticated API call

Overview

Every API integration starts with the right credentials.

Welcome to the Conga Advantage Platform. This guide walks you through creating an Integration User and obtaining your Client ID and Client Secret—the credentials you need for API access.

Conga recommends the Integration User approach over API Connections for most use cases. This guide covers both options.

Step 3
3 Add a new Integration User
Admin Console → Users → Add

Click the Add button. The User Details form appears, defaulting to the "Internal" user type. We need to change that.

The User Details form defaults to "Internal" user type

Click the User Type dropdown and select Integration. The form simplifies—Integration users don't need a Username or External ID.

Select "Integration" from the User Type dropdown
What is an Integration User?

An Integration user is a non-human system account designed for API access, data syncing, and third-party integrations. It authenticates via Client Credentials—no browser login required.

Step 4
4 Save and capture your Client ID and Secret

Click Save. A success toast appears: "The data is saved successfully" followed by "Credentials generated successfully." A Credentials popup then displays your two credentials:

Client ID

Your application's public identity. Safe to store in config files and environment variables.

Client Secret

Your application's private password. Guard it carefully.

The Credentials popup—copy both values immediately

Important

The Client Secret is shown only once. Once you dismiss this popup, the secret cannot be recovered. Copy it immediately and store it in a secrets manager or secure environment variable. If lost, you can click Generate Secret on the user’s detail page to regenerate credentials.

Store your secret in a secrets manager or secure environment variable.

Step 5
5 Make your first authenticated API call

With your Client ID and Secret, you can now authenticate via the OAuth 2.0 Client Credentials flow.

Step 5a—Request an access token

# Exchange credentials for an access token curl -X POST https://login-rls.congacloud.com/api/v1/auth/connect/token \ -H "Content-Type: application/x-www-form-urlencoded" \ -d "client_id=$CLIENT_ID" \ -d "client_secret=$CLIENT_SECRET" \ -d "grant_type=client_credentials" \ -d "scope=sign"

The scope parameter depends on which Conga API you're accessing:

ScopeAPI
signConga Sign
doc-gen.composerConga Composer
clmConga CLM (Contract Lifecycle)

Step 5b—Response

# You'll receive: { "access_token": "eyJhbGciOiJSUzI1NiIs...", "token_type": "Bearer", "expires_in": 3600, "scope": "sign" }
Step 5 (continued)

Step 5c—Call the API

Use the Bearer token from the response in all subsequent API requests:

# Use the Bearer token in subsequent requests curl https://api.congacloud.com/v1/your-endpoint \ -H "Authorization: Bearer $ACCESS_TOKEN" \ -H "Content-Type: application/json"
Token tip

Tokens expire after 3600 seconds (one hour). Your app should track expires_in and request a new token before it expires. Don't request a new token on every API call—reuse it.

That's all you need to start integrating with Conga APIs. The next page covers which regional endpoint to use, followed by appendices on regenerating credentials and an alternative access method.

Regional Endpoints
☉ API region endpoints

Conga operates in multiple regions. Use the Auth URL that matches your environment when requesting an access token, and the Base URL for subsequent API calls. Your region is determined by where your Conga organization was provisioned.

Production endpoints

RegionAuth URLBase URL
US login-rls.congacloud.com prod-rls10.congacloud.com
EU login.congacloud.eu rls.congacloud.eu
AU login.congacloud.au rls.congacloud.au

Preview (sandbox) endpoints

RegionAuth URLBase URL
US login-rlspreview.congacloud.com preview-rls09.congacloud.com
EU login-preview.congacloud.eu rls-preview.congacloud.eu
AU login-preview.congacloud.au rls-preview.congacloud.au
How to determine your region

Check the URL you use to log into the Conga platform. If it contains .congacloud.eu, you're in the EU region. If it contains .congacloud.au, you're in AU. Otherwise, you're in the US region. When in doubt, contact your Conga administrator.

For example, an EU production token request would use:

# EU production token request curl -X POST https://login.congacloud.eu/api/v1/auth/connect/token \ -H "Content-Type: application/x-www-form-urlencoded" \ -d "client_id=$CLIENT_ID" \ -d "client_secret=$CLIENT_SECRET" \ -d "grant_type=client_credentials" \ -d "scope=sign"

For full details, see documentation.conga.com → Conga Sign API Region Endpoints

Appendix A
! Regenerating a lost Client Secret

If you've lost your Client Secret, you can regenerate it. Here's how:

1. Navigate to Admin Console → Users

2. Find your Integration User in the list

3. Click the External Identifier hyperlink for that user

4. Click Save & Regenerate Client Secret

5. Copy the new Client ID and Client Secret from the popup

Important

Regenerating a secret invalidates the old one immediately. Update the new secret in all connected applications, or they will lose API access.
